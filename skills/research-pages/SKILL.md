# research-pages Skill

## このスキルがやること（ざっくり）

1. ユーザーが「このテーマについてリサーチしたい」と投げる
2. スキルが **一問一答（選択式＋短文）** で要件を整理する
3. 整理した要件から **Claude Code (MCP) 向けのリサーチプロンプト** を組み立てる
4. Claude Code MCP にリクエストを送り、リサーチ結果を受け取る（環境にMCP設定がある前提）
5. 結果を **1テーマ1ページ** のHTMLに整形する
6. ローカルの GitHub Pages リポジトリにファイルを書き出し、`git add/commit/push` を提案する
7. ユーザーの承認を得てから Git コマンドを実行し、公開URLを返す

このスキルは「何を調べるかまだフワッとしている状態」からでもスタートできるように、
最初に必ず一問一答で要件を固める。

---

## 前提・環境要件

- GitHub:
  - アカウントが存在する
  - GitHub Pages 用リポジトリ（例: `research-pages`）を作成予定 or 作成済み
- ローカル環境（OpenClawが動いているマシン）:
  - Git がインストール済み
  - GitHub Pages リポジトリをクローンしたローカルパス
    - 例: `~/Projects/research-pages`
  - OpenClaw からこのパス以下のファイルを読み書きできる
- Claude Code / MCP:
  - Claude Code を MCP 経由で呼び出せる環境設定がある前提
  - このスキルは「どういうペイロードで投げるべきか」を指示するだけで、
    実際のエンドポイントURLやトークンは環境依存とする

> 実環境の値をこのSKILL.mdの「前提」に追記してもよい。

## 典型的なフロー

### 1. テーマ入力

ユーザーが、例えば次のように言う:

> 「生成AIを使った個人の学習プロセス設計について調べたい」

スキルはすぐに直接リサーチを始めず、まず一問一答で要件を詰める。

### 2. 一問一答フェーズ（要件整理）

質問の例（必ず **一問ずつ** 出すこと）:

1. **用途はどれに近いか？**
   - A. 自分の勉強用
   - B. クライアント向け提案の下調べ
   - C. ブログ/記事用
   - D. その他（自由入力）

2. **想定読者は？**
   - A. 自分だけ
   - B. 身近な人（同僚・友人）
   - C. 不特定多数

3. **求める深さは？**
   - A. 概要レベル（5〜10分で読める）
   - B. 入門〜中級（20〜30分で読める）
   - C. ガッツリ専門寄り（論文・実装例にも触れる）

4. **特に知りたい観点は？（複数可だが、1問ずつ聞いていく）**
   - A. メリット・デメリット
   - B. 具体的なユースケース／事例
   - C. 実装・ツール選定
   - D. リスク・注意点

5. **ページの主なゴールは？**
   - A. 自分の頭の整理
   - B. 他人に見せる説明用資料
   - C. ブログ記事のたたき台

ユーザーの回答を、内部的な「リサーチ仕様オブジェクト」としてまとめる:

```yaml
topic: "生成AIを使った個人の学習プロセス設計"
purpose: "自分の勉強用"
audience: "自分だけ"
depth: "B"  # 入門〜中級
focus_points:
  - "具体的なユースケース"
  - "実装・ツール選定"
page_goal: "自分の頭の整理"
```

このオブジェクトを元に、Claude Code 向けプロンプトを組み立てる。

### 3. Claude Code 向けリサーチプロンプト生成

MCP 経由で Claude Code に送るべきプロンプトのイメージ:

```text
あなたはリサーチャー兼テクニカルライターです。

テーマ: 生成AIを使った個人の学習プロセス設計
用途: 自分の勉強用
想定読者: 自分自身（技術リテラシーは中級）
求める深さ: 入門〜中級（20〜30分で読めるボリューム）
フォーカスしたい観点:
- 具体的なユースケース
- 実装・ツール選定

出力フォーマット（Markdown）:

# タイトル

## 1. 概要
- テーマの一文要約
- このテーマが重要になる背景

## 2. ユースケース
- パターンごとに箇条書きで整理

## 3. 実装・ツール選定
- 代表的なツール/サービスの比較
- それぞれのメリット・デメリット

## 4. まとめ
- 今日から試せる一歩を3つ

## 参考リンク
- URLと短い説明

上記フォーマットに沿ってMarkdownで出力してください。
```

スキルはこのようなプロンプトを組み立てて、MCP経由でClaude Codeに送る。
MCPの呼び出し方法は環境依存のため、別途 `references/claude-code-mcp.md` などで指定する。

### 4. リサーチ結果 → HTML ページ化

Claude Code から Markdown が返ってきたら、次のステップを行う:

1. Markdown → HTML 変換（必要に応じてシンプルなパーサを使う想定）
2. 共通レイアウトに差し込む:
   - `<head>` にタイトル・メタ情報
   - `<main>` に本文
   - 最低限のCSS（assets/styles.css）を読み込む
3. ファイル名用の `slug` を生成:
   - 例: `"生成AIを使った個人の学習プロセス設計" → "ai-learning-process"`

HTMLの全体構造は `assets/base-template.html` を用意しておき、
`{{TITLE}}`, `{{CONTENT}}` などのプレースホルダを置き換える形で実装する。

### 5. GitHub Pages リポジトリへの書き込み

前提として、ローカルに GitHub Pages リポジトリがある:

- 例: `~/Projects/research-pages`
- ページ置き場: `~/Projects/research-pages/research/`

スキルは次を行う:

1. `research/<slug>.html` パスにHTMLを書き出す
2. `git status` で差分を取得してユーザーに提示
3. 以下のようなコマンドのプランをテキストで示す:

   ```bash
   cd ~/Projects/research-pages
   git add research/<slug>.html
   git commit -m "Add research page: <topic>"
   git push
   ```

4. 「このコマンドを実行していい？」とユーザーに承認を求める
5. 承認されたら、OpenClawの `exec` ツールなどを使って順番にコマンドを実行する

### 6. 公開URLを返す

リポジトリが GitHub Pages に設定されている前提で、公開URLは例えば:

```text
https://<username>.github.io/research-pages/research/<slug>.html
```

スキルは `<username>` とリポジトリ名を前提から組み立て、
ユーザーに最終的なURLを教える。
