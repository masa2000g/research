# 梓モーニングコールシステム 要件定義 / 仕様書（ドラフト）

## 1. 概要

- **目的**: 朝の決まった時間に、梓（AIお姉さん）が電話でマイクを起こし、軽い会話とモーニングルーティンへの誘導を行う。
- **手段**:
  - Twilio Programmable Voice を使って電話を発信
  - Node.js サーバーで通話フローを制御
  - 将来的に STT / LLM / TTS と連携して対話型に拡張

本ドキュメントは **フェーズ1〜2** にフォーカスし、
- フェーズ1: 電話発信 + 固定メッセージ再生
- フェーズ2: モーニングコール向けの簡易会話フロー（パターンD）

までをカバーする。

---

## 2. スコープ

### 2.1 対象フェーズ

1. **フェーズ1（最小動作）**
   - HTTPエンドポイント `/start-wakeup-call` を叩くと、Twilio 経由でマイクのスマホに電話をかける。
   - マイクが電話に出ると、梓の固定メッセージ（日本語）が再生される。
   - 会話（音声入力）は扱わない。通話はメッセージ再生のみ、数秒で終了。

2. **フェーズ2（会話の型を導入）**
   - モーニングコール向けに、以下の会話パターンDをベースにシナリオを固定テキストで実装。
   - 将来的な STT / LLM 連携を見越しつつも、このフェーズではまだ外部LLM連携は行わない（テキストは固定、または簡単な分岐のみ）。

> パターンD（ベース会話）
>
> 1. 梓: 「おはよう、マイク。ちゃんと電話取ってえらい。今の気持ち、正直に言うとさ、“まだ寝てたい”と“起きなきゃ”どっちが勝ってる？」
> 2. マイク: 「まだ寝たい…」「起きなきゃ…」など
> 3. 梓: 「まだ寝たい派か〜。じゃあさ、“あと○分だけ”って自分で決めよ？その代わり、そのあと起きたら、いつもの“朝レポートタイム”だけ一緒にやろ？調査レポートとか、気になってる記事をひとつ読むところまでをゴールにしよ。」

## 4. システム構成（現状案）

### 4.1 コンポーネント

- **OpenClaw (Azusa)**
  - 朝のcronジョブ管理（5:00メッセージ、5:15追いかけメッセージ等）
  - 起床状況に応じて `/start-wakeup-call` を叩く役

- **Node.js サーバー（本プロジェクト）**
  - Express ベースの API サーバー
  - エンドポイント:
    - `POST /start-wakeup-call`: モーニングコール開始
    - `POST /twiml/wakeup`: Twilio が通話内容(TwiML)を取得するエンドポイント
  - Twilio Node.js SDK を使用

- **Twilio Programmable Voice**
  - マイクの電話番号への発信
  - `/twiml/wakeup` から取得した TwiML に従って、音声合成 / 再生を行う

### 4.2 デプロイ想定

- 開発初期:
  - Mac mini 上でローカル起動 (`node index.js`)
  - 必要に応じて ngrok 等で外部公開
- 将来:
  - 任意のVPS / PaaS（Render, Railway, Cloud Run, Fly.io など）にデプロイ

## 5. 環境変数 / 設定値

`.env` ファイルで管理する想定（dotenv使用）。

- `TWILIO_ACCOUNT_SID`: Twilio アカウント SID
- `TWILIO_AUTH_TOKEN`: Twilio Auth Token
- `TWILIO_CALLER_NUMBER`: 発信用Twilio電話番号（E.164形式）
- `MASA_PHONE_NUMBER`: マイクのスマホ番号（E.164形式）
- `PORT` (任意): HTTPサーバーポート（デフォルト: 3000）

`.env.example` を用意し、サンプル値を記載する。

## 6. エンドポイント仕様

### 6.1 `POST /start-wakeup-call`

- **概要**: モーニングコール発信を開始する。
- **想定呼び出し元**: OpenClaw / 手動テスト用の curl / Postman
- **リクエストボディ**: 現時点では必須パラメータなし。将来的に:
  - `scenario`: シナリオ名（例: `"morning_pattern_d"`）
  - `delayMinutes`: 何分後にかけるか（即時発信でなければ）

- **レスポンス (成功時)**:
  - ステータス: `200 OK`
  - Body:
    ```json
    {
      "ok": true,
      "sid": "CAxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
    }
    ```

- **レスポンス (失敗時)**:
  - ステータス: `500 Internal Server Error`
  - Body:
    ```json
    {
      "ok": false,
      "error": "エラーメッセージ"
    }
    ```

### 6.2 `POST /twiml/wakeup`

- **概要**: Twilio が通話開始時にアクセスし、何を喋るか(TwiML)を取得する。
- **想定呼び出し元**: Twilioサーバー
- **リクエストボディ**: Twilio標準のWebhookパラメータ（CallSid, From, To など）
- **レスポンス**:
  - Content-Type: `text/xml`
  - Body: TwiML

例（フェーズ1固定メッセージ）:

```xml
<Response>
  <Say voice="alice" language="ja-JP">
    おはよう、マイク。梓だよ。まだ寝てたい気持ちもわかるけど、そろそろ起きよっか。
  </Say>
</Response>
```

フェーズ2以降では、ここでシナリオに応じたメッセージや将来的な会話ハンドリングへの分岐を行う。

## 7. ロジック概要（フェーズ1）

1. `/start-wakeup-call` が呼ばれる。
2. Node.js サーバーは Twilio SDK を使って `client.calls.create(...)` を実行。
3. `url` パラメータとして `/twiml/wakeup` の公開URLを指定。
4. Twilio がマイクの電話番号に発信。
5. マイクが電話に出ると、Twilio は `/twiml/wakeup` にHTTPリクエストを送る。
6. サーバーは固定メッセージを含む TwiML を返却。
7. Twilio が合成音声でメッセージを再生し、通話終了。

## 8. ロジック概要（フェーズ2構想 / メモ段階）

- 現段階では**実装は行わず、設計メモのみ**残す。

### 8.1 目標

- パターンDをベースに、以下を実現する:
  - 梓が「まだ寝たい vs 起きなきゃ」を聞く
  - マイクの返答（将来的には音声→テキスト）に応じて分岐
  - 「あと○分だけ寝て、そのあと朝レポートタイムをやる」という合意形成

### 8.2 想定アプローチ

- Twilio の `<Gather>` を使って、DTMF入力や短い音声入力をトリガーにする
- フルSTT+LLMまでは一気に行かず、最初は「1 or 2の選択」など簡易な分岐を試す
- STT/LLM 連携フェーズでは:
  - 音声 → STT（Whisper/Deepgramなど）
  - テキスト → LLM（梓プロンプト）
  - 返答テキスト → TTS（Twilio TTS or 外部TTS）
